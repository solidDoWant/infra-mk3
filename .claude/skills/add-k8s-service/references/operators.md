# Operators, Backends, and Storage

---

## Storage Class Selection

| Class | When to use |
|-------|-------------|
| `ceph-block` | **Default for application persistent data.** Block devices move between nodes with pods. Use for app state, anything that needs reliable RWO access. |
| `nfs` (volume type) | Large data pools where slower access is acceptable (media files, large archives). Use `${NFS_ADDRESS}` and `${NFS_MEDIA_PATH}` substitution variables. |
| `ceph-filesystem` | **Only** when multi-writer (RWX) access is a hard requirement **and** the application explicitly documents CephFS support. Always flag when proposing this — it has additional overhead and complexity. |
| `zfs-generic-nfs` | Backups only. Used specifically because it supports the volume snapshot API, which plain NFS does not. Moves the failure domain outside the cluster. |
| `zfs-openebs` | Only when the application itself provides data replication (e.g., CNPG manages its own replication). In that case, create a **dedicated storage class** with tuned performance options for this specific workload — don't use a shared class. |
| `local-path` | **Never.** Pods become permanently bound to a single node. |

**Summary**: Use `ceph-block` by default for application data. Use NFS for media/large shared data. Only use the others when there's a hard requirement.

**Exceptions — do not use `ceph-block` for:**
- **PostgreSQL**: Uses a dedicated `zfs-openebs` storage class (configured by the `cluster/charts/postgres/cluster` chart). CNPG manages its own replication, so local ZFS storage is correct — Ceph replication would be redundant.
- **Dragonfly**: Does not use persistent storage at all. It is an in-memory cache; data is ephemeral by design.

## Envoy Sidecar (for apps that don't support backend mTLS natively)

When an app connects to postgres or dragonfly but can't perform TLS client auth itself, use Envoy as a transparent proxy sidecar.

**How it works:**
1. Envoy runs as an `initContainer` with `restartPolicy: Always` (Kubernetes sidecar pattern)
2. Envoy listens on `127.0.0.1:<port>` (same port as the real backend)
3. App connects to `127.0.0.1:<port>` with no TLS — Envoy upgrades to mTLS upstream
4. Envoy config is stored in a `ConfigMap` generated by kustomize

**Reference implementation:** `cluster/gitops/media/bazarr/app/` (postgres + file-based SDS hot-reload).

**Reusable Envoy configs:** copy all files for the chosen backend into `app/envoy-config/` — the cluster envoy.yaml references the SDS YAMLs by filename at `/etc/envoy/...-sds.yaml`, so all three must land in the same configmap.

PostgreSQL:
- `references/envoy-sidecar-pg.yaml` — main envoy config
- `references/envoy-sidecar-pg-client-cert-sds.yaml` — SDS resource for client cert
- `references/envoy-sidecar-pg-validation-context-sds.yaml` — SDS resource for the root CA

Dragonfly/Redis:
- `references/envoy-sidecar-redis.yaml` — main envoy config
- `references/envoy-sidecar-redis-client-cert-sds.yaml` — SDS resource for client cert
- `references/envoy-sidecar-redis-validation-context-sds.yaml` — SDS resource for the root CA

**Important notes:**
- Always use the `contrib` Envoy image — `envoyproxy/envoy:contrib-vX.Y.Z` — the postgres filter extension is only in contrib builds. Always use the latest available version.
- For Dragonfly mTLS, use `references/envoy-sidecar-redis.yaml`. This works for most apps. The exception is apps that use multiple Redis database indexes (`SELECT N`) — the redis_proxy filter doesn't support this (envoyproxy/envoy#41659). Check whether the app uses multiple indexes before recommending this path.
- Set `idle_timeout: 0s` in the TCP proxy filter — let the application manage idle connections rather than Envoy terminating them (some apps handle mid-connection resets poorly).
- **Set `node.id` and `node.cluster`** at the top of the envoy config. Envoy's SDS code requires both even with file-based `path_config_source` (it fills them into the discovery request); without them the container fails to start with `node 'id' and 'cluster' are required`. Values are arbitrary — they only appear in stats/logs. Use `envoy-<service>-<backend>-proxy` and `<namespace>`.
- **Cert hot-reload via SDS**: the envoy configs use `tls_certificate_sds_secret_configs` and `validation_context_sds_secret_config` with `path_config_source.watched_directory` pointing at the cert mount directories. When cert-manager rotates a cert, k8s atomically swaps the projected Secret's `..data` symlink; Envoy's inotify on the mount picks that up and reloads the cert without a pod restart. This requires the postgres chart's `users.<service>.disableReloading: true` (default in the skill's reference template) — otherwise the cnpg.io/reload label on the cert secret would cause kyverno to trigger a pod restart anyway, defeating the hot-reload.

### Envoy container (for app-template `initContainers`)

```yaml
controllers:
  <service>:
    initContainers:
      envoy:
        image:
          repository: envoyproxy/envoy
          tag: contrib-v1.35.3      # Always pin; use latest available
        ports:
          - name: envoy-admin
            containerPort: 8000
        securityContext:
          capabilities:
            drop: [ALL]
          privileged: false
          runAsNonRoot: true
          allowPrivilegeEscalation: false
          seccompProfile:
            type: RuntimeDefault
        restartPolicy: Always       # Required for sidecar container pattern
```

### For external charts (via postRenderers)

When the chart doesn't expose `initContainers` in values, inject via `postRenderers.kustomize.patches`. See `cluster/gitops/development/harbor/app/hr.yaml` for the full patch syntax — it injects both the container spec and the required volumes onto specific Deployments/StatefulSets by name.

### Envoy config ConfigMap structure

Store envoy config YAML files in `app/envoy-config/` and generate a single ConfigMap that contains both the main envoy.yaml and the SDS resources (they live side-by-side at `/etc/envoy/` in the container):

```yaml
# app/envoy-config/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization
configMapGenerator:
  - name: <service>-envoy-pg-config
    files:
      - envoy.yaml=./postgres.yaml
      - postgres-client-cert-sds.yaml=./postgres-client-cert-sds.yaml
      - postgres-validation-context-sds.yaml=./postgres-validation-context-sds.yaml
    options:
      labels:
        kyverno.home.arpa/reload: "true"   # Pod restart on envoy config / SDS YAML change
      disableNameSuffixHash: true
```

The `kyverno.home.arpa/reload` label is still required on the ConfigMap because Envoy doesn't hot-reload its top-level config or SDS file references — only the cert contents referenced by SDS hot-reload. Top-level config changes (e.g. changing the postgres host, adding a listener) need a pod restart, which the label triggers via the workload-reloader policy.

### Required volumes (app-template persistence)

The client cert and root CA **must** be mounted at **separate directories** (not co-located with `subPath`). SDS hot-reload relies on inotify on the `..data` symlink swap, which doesn't propagate to `subPath` file mounts. Each watched directory is a complete Secret mount.

```yaml
persistence:
  envoy-config:
    type: configMap
    name: <service>-envoy-pg-config
    defaultMode: 0440
    advancedMounts:
      <service>:
        envoy:
          - path: /etc/envoy
  postgres-cert:
    type: secret
    name: <service>-postgres-<user>
    defaultMode: 0440
    items:
      - key: tls.crt
        path: tls.crt
      - key: tls.key
        path: tls.key
    advancedMounts:
      <service>:
        envoy:
          - path: /etc/envoy/secrets/postgres
  root-ca:
    type: secret
    name: root-ca-pub-cert
    defaultMode: 0444
    items:
      - key: ca.crt
        path: ca.crt
    advancedMounts:
      <service>:
        envoy:
          - path: /etc/envoy/secrets/root-ca
```

## SMTP Relay (docker-postfix)

Not an operator — just a shared backing service already deployed in the `email` namespace.

Configure apps to use it directly:
- Host: `docker-postfix.email.svc.cluster.local`
- Port: `587` (STARTTLS)
- From address: `${SECRET_EMAIL_ADDRESS}`

Add egress to the network policy:
```yaml
# Allow SMTP relay access for sending emails
- toEndpoints:
    - matchLabels:
        app.kubernetes.io/name: docker-postfix
        io.kubernetes.pod.namespace: email
  toPorts:
    - ports:
        - port: "587"
          protocol: TCP
```

## Container Registry Preference

Prefer non-Docker Hub registries (ghcr.io, quay.io, registry.k8s.io, public.ecr.aws) over docker.io. Docker Hub anonymous pulls are rate-limited and add a single point of failure — pulling from GHCR/Quay/etc. avoids both.

Selection order:

1. **home-operations** — `ghcr.io/home-operations/<service-name>`. Browse at https://github.com/home-operations/containers. These community-maintained images typically:
   - Run without root
   - Have stronger security controls (minimal base images, no unnecessary packages)
   - Are maintained in lockstep with upstream releases
2. **Upstream project's own non-Docker-Hub registry** — most projects publish to `ghcr.io/<org>/<name>` or `quay.io/<org>/<name>` (often in addition to Docker Hub). Check the project's README, release notes, or GitHub Packages tab before falling back. The image is usually byte-identical to the Docker Hub one.
3. **docker.io** — only when no non-Docker-Hub source exists for the image.
